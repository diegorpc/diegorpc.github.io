Starting your chat app

Actions (What, Where, Who):

1. Create Chat

What: Create a new chat with a specific title and invite members. This is represented with a Graffiti object with the following value:

{s
    activity: "Create",
    type: "Chat",
    title: "My chat title",
    channel: crypto.randomUUID(), // The "Where" for sending chat-specific messages
    members: ["did:plc:abc123...", "did:plc:def456..."], // Actor IDs of invited members
    published: Date.now() // timestamp
}

Where: New chats are stored in the creator's personal channel ("[actorID]/chats") so they can track their own chats. The chat object is also posted to each invited member's inbox channel ("[memberActorID]/inbox") to notify them of the invitation.

Who: The chat creation object should only be visible to the invited members. The allowed list contains the actor IDs of the chat creator and all invited members.

2. Send Message

What: Send a message in a specific chat. This is represented with a Graffiti object with the following value:

{
    activity: "Create",
    type: "Message",
    content: "Hello everyone!",
    target: "d222491b-d6d0-4c21-aadd-45ed9bc21566", // The chat channel
    published: Date.now() // timestamp
}

Where: Messages are stored in the specific chat's channel (the UUID generated when the chat was created) so they can be discovered by members of that chat.

Who: Messages are private and only visible to chat members. The allowed list contains the actor IDs of all chat member.

3. Join/Accept Chat Invitation

What: Accept an invitation and join an existing chat. This is represented with a Graffiti object with the following value:

{
    activity: "Join",
    type: "Chat",
    target: "d222491b-d6d0-4c21-aadd-45ed9bc21566" // The chat channel
}

Where: The join action is stored in two places: (1) the chat's channel so other members can see who joined, and (2) the user's personal channel ("[actorID]/chats") so they can track which chats they're in.

Who: In the chat's channel, the join action is visible to all chat members. In the user's personal tracking channel, it's stored with an empty allowed list (only visible to them).

4. Set Profile

What: Set or update your display name and profile informtion. This is represented with a Graffiti object with the following value:

{
    type: "Profile",
    name: "Diego Peon",
    describes: "did:plc:zd7szk2575bygcrlt2m35zj6" // The actor ID of the person
}

Where: Profiles are stored in the user's personal channel ("[actorID]/profile") so they can be discovered by others who need to display the user's name.

Who: Profiles are intended to be public (no allowed list) so anyone can see the display names of users they interact with.

Other Design Decisions:

  • From our implementations in studio, I was particularly mindful of loading states. All async operations (creating chats, sending messages, loading chats) display loading indicators to provide feedback which are passed through signals.
  • The app uses a single-screen layout with a scrolling message container and a sticky footer for the message input as I had originally mocked up in paper and Figma. This is clearly mobile-first but actually translated quite well to desktop with some slight changes to the styling (while still keeping the core identity of my prototypes).
  • Chats have multiple channels for each user. I figure this approach is appropriate for including multiple people in a chat but let me know if a shared channel is sufficient and we just control users through the allowed list.
  • I still haven't implemented any of my novel features in the UI like pages and broadcast messages, focusing just on the chatting/chat creation the exercise indicates.
